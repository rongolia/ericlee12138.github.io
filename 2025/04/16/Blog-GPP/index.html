Observer

  • The pattern behind MVC
  • Pretty much the same as Event System
  • In highly threaded engine, do asynchronous communication with Event Queue
  • Dynamic allocations take time
    • Linked observers
      • SUBJECT and OBSERVER should be friends
      • AddObserver to the end adds complexity, while AddObserver to the front has one side effect, where newly added observer will get notified first
      • RemoveObserver usually requires special work for removing the first one. Doing pointer to pointer can overcome this
      • One observer can only observer one subject
    • Linked list with pointer to observers
    • Object Pool?
  • Remaining problems
    • Destroying subjects/observers
      • It’s an observer’s job to unregistered itself when it gets deleted
      • Have an subject send one final “dying” notification before getting destroyed
      • Or make observers automatically unregister themselves from every subject when they get destroyed. Then each observer needs a list of all the subjects → pointers going both directions
    • Garbage collectors
      • The lapsed listener problem
  • If you often need to think about both sides of some communication in order to understand a part of the program, don’t use the Observer pattern to express that linkage. Prefer something more explicit.
  • The observer pattern is a great way to let those mostly unrelated lumps talk to each other without them merging into one big lump. It’s less useful within a single lump of code dedicated to one feature or aspect.
  • Even in C++, I would tend toward a system that let you register member function pointers as observers instead of instances of some Observer interface.

Prototype

Different monsters & monster spawners Lots of classes and duplication
→ Abstract clone function in base class and handle spawning in child classes Also cloning states, yet still lots of classes
→ Pass spawn function, template No spawner class
  • In most OOP, state is contained in the instance, behavior is contained in the class (v-table)
  • Self → JavaScript
  • Prototype is useful for data modeling
    • No need for an abstract prototype, just pick the one that’s the simplest

Singleton

To solve two problems - restricting a class to one instance & global access point

Why we use it

  • Does not create the instance till needed
  • Initialized at runtime
  • Can be subclassed

Why we regret using it

  • Global variable
    • Hard to debug - anyone can touch the global state
    • Encourage coupling - anyone can access it can couple it into their code
    • Not concurrency friendly
  • Sometimes I only have one of the two problems
  • Lazy initialization takes control away

What to do

  • Remove the class if possible - OOP is about letting objects take care of themselves
  • Make access easy but not global
    • Pass the object in
    • Get it from there base class
    • Get it from something already global
    • Service Locator

State

Error prone - complex branching and mutable state

FSM

  • Fixed set of states
  • In one state at a time
  • External inputs/events
  • Each state has a set of transitions

State interface → state classes → delegate

State objects - static or instantiated

Exit and enter actions - let the switching-to state instead of the switching-from state handle graphics changes

Concurrent state machine

Use two separate FSMs for two different sets of states

When the two FSMs are interacting

  • Use some if statement
  • Have one state machine consume the input

Hierarchical state machine

Event → sub state → super state → …

Implement - class inheritance or a stack of states

Pushdown automata

History states

Modern: behavior tree & planning systems

Double Buffer

When to use

  • State being modified incrementally
  • State possibly being accessed during modification
  • Want to prevent the accessing code from seeing the work in progress
  • Want to read the state without having to wait

Remember

  • Swapping buffers takes time - pointless if swapping takes more time than modifying
  • Two buffers - memory usage

How to swap buffers

  • Swap pointers or references
    • Fast
    • External code cannot store persistent pointers to the buffers - main limitation, can be an issue if the buffers are expected to be at a fixed location
    • Existing data on the buffer are from two frames ago - can be an issue if we need to reuse data from last frame, e.g. motion blur
  • Copy data
    • Existing data on the buffer are one frame old
    • Swapping can take a while, unless the state is a single flag

Game Loop

Simplest

No control over game speed, fast on fast machine, crawl on slow machine

Sleep

Make sure the game doesn’t run too fast, doesn’t help when it’s slow

A bit more sophisticated solution

Pass in how much real time has passed

Consistent frame rate

Faster machine gets smoother gameplay

Non-deterministic and unstable, different frame rates can have different game results, physics gets unstable

Rendering doesn’t care about elapsed time

MS_PER_UPDATE should be pretty short, but not too short or it won’t catch up

Yank rendering out of the update loop to free up some CPU time

Rendering right between two frames

Extrapolate based on game object position and velocity, it can be wrong but less noticeable than the stuttering

Update

Splitting code into frames is more complex but almost always necessary

Have to store data to resume

  • State might help

All objects simulate but not truly concurrent

  • Orders matter, former objects see latter objects’ old state, latter objects see former objects’ new state

Be careful modifying the object list

  • Adding a new object will have it update during the same frame it spawns - cache the number of objects to avoid
  • Removing an object can cause skipping objects
    • Mark them dead and remove them when done walking the list
    • Walk the list backwards

Entity class

  • Class inheritance - terrible, use Component
  • Passing elapsed time

Delegate class

  • Used in State and Type Object

Dormant objects

  • Single collection containing inactive objects - waste time on checking flag or calling an empty method
  • Separate collection - extra memory and need to keep collections in sync

Bytecode

A small virtual machine

Structure

  • APIs
  • Instruction set - one byte hence the name
  • Stack machine

Literal instruction

For disambiguation when reading constants

Other approach - constant pool

Stack-based or register-based

Stack-based PUSH 2, PUSH 3, ADD

  • Instructions are small, arguments are always on top of the stack
  • Code generation is simpler
  • There will be more instructions

Register-based ADD R0, R1, R2

  • Instructions are large, lots of arguments
  • There will be fewer instructions

What instructions

  • External primitives, actual game behaviors
  • Internal primitives, manipulate internal values, e.g. literal, arithmetic, comparison, etc.
  • Control flow, conditions or loops, similar to goto
  • Abstraction, reusable callable procedures, return stack

What values

  • Single data type
    • Simple, both good and bad
  • Tagged union, enum + union → most common
    • Values know their type
    • More memory
  • Untagged union
    • Compact, fast
    • Unsafe, misinterpreting a value can cause crash
  • Interface, class and subclasses
    • Open-ended, easy to define new value type
    • Object-oriented
    • Redundant, a separate class for each value type
    • Inefficient

What scripts

  • Text-based
    • Syntax required
    • Parser required
    • Error handling required
    • Not friendly to untechnical users
  • Graphical tool
    • GUI required
    • Fewer error cases
    • Hard portability across systems

Subclass Sandbox

Lots of subclasses

When you find yourself with a lot of subclasses, it often means a data-driven approach it better.

  • Redundant code
  • Game engine coupled to the classes and changes to outside systems will get the classes broken
  • Hard to define invariants

Pattern

Base class = abstract sandbox method + provided operations

Subclasses = implement sandbox method using the provided operations

When to use

  • Base class and a number of subclasses
  • Base class can provide all operations needed by subclasses
  • Behavioral overlap among subclasses
  • Minimize coupling between subclasses and other systems

If the base class is turning into a big code stew, consider pulling some provided operations into separate classes, with Component

What provided operations

NO provided operations, subclasses call into outside systems ↔ EVERY operation needed, subclasses are ONLY coupled to the base class

  • A provided operation only used by very few subclasses - not very beneficial, let special case subclasses call into external systems

  • A method not modifying any state is “safe” coupling. Methods that do modify state are good candidates for provided operations

    Even just accessing data can cause problem, e.g. in a multi-threaded systems, reading something while it’s being modified

  • A provided operation only forwards a call to external systems - may be simpler to directly call, but it can still be useful

How are operations provided

Directly - lots of methods

Indirectly - helper classes, fewer methods, help classes are easier to maintain, lower the coupling between the base class and external systems

How does base class get state

  • Pass through constructor - exposing the hidden state to subclasses
  • ctor + init - need to make sure init is called
  • Make the state static - works for singleton systems, can have good and bad of singleton
  • Service Locator

Update method is often also a sandbox method. Subclass Sandbox is a role reversal of Template Method
